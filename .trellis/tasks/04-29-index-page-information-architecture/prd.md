# Improve index page information architecture

## Goal

Improve the `site/index.html` overview page so it works better as a browsing and management surface for project insight reports. The page should remove redundant summary content, make the report list easier to scan, and use search plus tags as the primary discovery model instead of forcing projects into rigid `类型` / `采用口径` dropdown categories.

## What I already know

* The user provided screenshots showing the current report list and toolbar.
* The hero metric `1 / 统一列表视图，面向浏览和管理` is redundant and does not help users choose a report.
* The report list currently mixes title, slug, adoption, tags, four detail fields, and three action buttons in a dense row. The result is visually noisy and hard to compare.
* The toolbar currently exposes `类型` and `采用口径`, but the underlying values in `site/projects.json` mix product type, technical form, audience, and readiness language without a clear taxonomy.
* `site/index.html` is generated by `tools/generate_project_index.py`; generated HTML should not be hand-edited without updating the generator.
* Index filtering and rendering are plain embedded JavaScript, not a framework app.
* After reviewing the first taxonomy implementation, the user decided fixed `类型` / `采用口径` dropdowns are likely less useful than tags plus keyword search.
* After reviewing the tag-first implementation, the user also found the standalone `项目形态` metric and per-row shape/adoption badges unstable. These dimensions should be folded into tags instead of displayed as separate stable categories.
* The user reviewed current page copy and found it still too maintenance-oriented. The index should read like a report browsing surface, not a generator/admin page.
* The user reviewed the current hero area and found the top `Project Insight Reports / N 个项目 · 更新至...` line low value because the hero and metrics already communicate page identity and status.
* The user wants important phrases inside the report list summary/detail text to have visible emphasis instead of all facts rendering at the same weight.
* After reviewing the emphasized list, the user found the page still visually tiring because too many emphasized phrases share the same black weight and color. The list needs stronger visual hierarchy, not just more bold text.
* After reviewing the colored detail rows, the user found the color usage still too strong. The row hierarchy should stay visible, but color should be restrained and secondary to typography/spacing.
* The user wants the report-row detail area optimized like a designed graphic rather than a stack of colored bars.

## Assumptions

* "现实" in the request means the index page's display / presentation.
* This task should improve the existing static index page, not rebuild the site in React/Vue or introduce a backend.
* The first implementation should keep the current `site/projects.json` file as the data source.
* The taxonomy can be made explicit with stable display labels and short descriptions without adding a database.
* Because the project list is still small, it is acceptable and preferred to normalize `site/projects.json` directly instead of hiding all taxonomy cleanup in generator-only mappings.
* `category` and `adoption` should not be standalone display dimensions. If kept in data, they are compatibility/search metadata only; the visible UI should use tags.

## Requirements

### Remove redundant overview content

* Remove or replace the hero-side metric that currently says `1 / 统一列表视图，面向浏览和管理`.
* Remove the separate top line above the hero that repeats `Project Insight Reports` and `N 个项目 · 更新至...`; keep identity and metrics inside the hero area.
* Keep only overview information that helps browsing or maintenance, such as total projects, category count, last update, or quality/status hints.
* Avoid explanatory text that repeats obvious UI behavior.

### Rewrite reader-facing copy

* The page is for report readers, not maintainers.
* Hero eyebrow should be `Project Insight Reports`.
* Hero title should be `项目洞察报告`.
* Hero supporting copy should communicate the value of multi-dimensional project analysis, not just search mechanics. Use: `从问题场景、目标用户、差异机制、架构视角和采用风险等维度理解一个项目。`
* Report-list supporting copy should explain what the reports contain and how to narrow them. Use: `每份报告围绕问题、差异、机制、架构和采用判断展开；可用关键词或标签收窄列表。`
* Remove maintenance/generator language from the main page body, including `新增项目只维护清单数据`, `扩充：`, and command-line instructions.
* Remove the standalone slug subtitle under project titles, such as `andrej-karpathy-skills`; the title and GitHub action carry enough identity for readers.
* Under each project title, show a one-sentence functional highlight / capability summary, not the target audience.
* The functional highlight should be explicit data, not inferred from `audience`. Add a `summary` field to `site/projects.json` if needed.

### Improve report list layout

* Redesign each report row/card so the user can scan:
  * project title
  * one-sentence functional highlight
  * the primary problem solved
  * the main difference or demo mechanism
  * actions
* Reduce visual clutter from too many equal-weight fields.
* Make action buttons visually consistent and easy to find without dominating the row.
* Preserve all three current destinations: report page, Markdown report, GitHub repository.
* Keep mobile layout readable with no horizontal overflow.
* Support lightweight inline emphasis for important phrases in report list text, using the existing Markdown-style `**...**` / code-span convention and safe escaping before injecting HTML.
* Use distinct visual treatments for the list facts so users can find priority quickly:
  * `解决问题` should read as the primary scan target.
  * `差异点` and `Demo / 机制` should remain visible but lower than the main problem.
  * Emphasized phrases should use restrained accent colors tied to their row type instead of all rendering as heavy black text.
* Keep color usage restrained: avoid broad saturated color blocks; prefer subtle rails, very light neutral fills, and dark neutral emphasis text.
* Present the three detail facts as a cohesive analysis group, with quiet separators and small type markers instead of full-width colored bands.

### Shift discovery to search + tags

* Remove `类型` and `采用口径` dropdowns as primary filters.
* Keep one prominent search box that searches title, slug, repo, problem, difference, demo, architecture, category, adoption, and tags.
* Add visible tag chips as a lightweight discovery layer.
* Clicking a tag chip should filter/search by that tag or otherwise narrow results in a predictable way.
* Use tags for multi-dimensional concepts such as `Agent`, `MCP`, `CLI`, `API Proxy`, `Knowledge`, `Desktop`, `Finance`, `本地试点`, `团队基线`, and `需审计`.
* Fold `category` and `adoption` concepts into the tag set instead of rendering them as separate badges, metrics, or dropdowns.
* Remove the hero metric/card that counts or names `项目形态`.
* Remove per-row category/adoption badges such as `接口代理` and `本地试点`; represent those concepts through tags if still useful.
* If `site/projects.json` tags are expanded for discoverability, regenerate the index and keep the search behavior consistent.

### Preserve current static-site architecture

* Update `tools/generate_project_index.py` as the source of truth.
* Regenerate `site/index.html`.
* Update `tools/verify_reports.js` if selectors, counts, or expected index behavior change.
* Do not introduce a framework, bundler, database, or API server.

* Adding new reports.

## Technical Notes

* Primary files likely affected:
  * `tools/generate_project_index.py`
  * `site/projects.json`
  * `site/index.html`
  * `tools/verify_reports.js` if selectors or layout expectations change
* Before this task, the index generator computed categories from raw `project.category` values and adoption filter values from raw `project.adoption` values.
* The updated row renderer keeps `.project-row`, `.project-title`, `.actions`, and `.tag-row`, and uses `.project-summary` plus `.summary-stack` / `.summary-line` for the report analysis facts.
* The updated verifier checks the tag-first discovery model, inline emphasis rendering, absence of category/adoption controls and badges, index links, and `gbrain` search behavior.

## Decisions

* Normalize `site/projects.json` directly. The current project count is small, so direct data cleanup is simpler and clearer than maintaining a hidden generator-only taxonomy mapping.
* Replace strict category/adoption dropdown discovery with search plus tag chips. Tags are a better fit for this report directory because projects are multi-dimensional and users are more likely to search for concrete terms than choose a rigid taxonomy first.
* Do not display `项目形态` / `category` or `采用口径` / `adoption` as standalone stable dimensions. Keep these ideas in tags instead.
* Treat the page as a reader-facing report directory. Maintenance commands belong in README or project docs, not in the index page body.
* Do not use `audience` as the title-area subtitle in report rows. Use explicit functional highlight copy instead.
