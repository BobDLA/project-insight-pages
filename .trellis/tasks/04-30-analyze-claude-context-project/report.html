01

新用户先看什么

适合谁、解决什么问题、差异、时机和最小验证。

适合谁

正在用 Claude Code、Cursor、Gemini CLI、Codex CLI 等 agent 工具，且仓库大到 grep/read 多轮探索成本高的团队。

解决什么问题

把“找相关代码”从临时上下文消耗变成可复用索引：agent 用自然语言搜索，返回文件路径、行号和片段。

和别的方案哪里不同

主入口是 MCP 工具；默认 hybrid search；同时处理异步索引、部分结果、snapshot、Merkle 增量同步和触发 watcher。

为什么现在值得看

项目仍在活跃迭代，源码里有多处生产化边缘处理，且 evaluation 给出了可复现实验假设。

最小验证方式

选内部真实维护任务，对比 grep/read 与 grep/read + `search_code` 的 token、工具调用、命中速度和误命中率。

先不适合的场景

不允许代码片段进入外部 embedding 服务、不能维护向量库、或只需要轻量本地字面搜索的场景。

Gold Example / Demo

Demo 状态：静态推演，未运行；示例来自 README 与 MCP/Context 源码。

最小闭环

  • 配置 MCP：`npx @zilliz/claude-context-mcp@latest`，提供 embedding 与 Milvus/Zilliz 参数。
  • 用户说“Index this codebase”，MCP 写入 `indexing` 状态并后台执行索引。
  • 用户查询状态，读取本地 snapshot 的进度、失败或完成统计。
  • 用户自然语言搜索，返回代码片段、路径和行号。

为什么这个例子重要

它体现项目真正卖点：不是单次 embedding API 调用，而是 agent 可以持续调用的代码上下文层。README 的效率数据是有用假设，但本报告未复现实验。

05

核心资产与价值

MCP 工具入口

`packages/mcp/src/index.ts` 与 `handlers.ts` 把索引、搜索、状态和清理变成 agent 可调用工具。

Core Context

`packages/core/src/context.ts` 聚合文件发现、ignore、分块、embedding、collection、search 和 dedup。

AST + fallback splitter

Tree-sitter 分块覆盖多语言，失败或不支持时回退 LangChain splitter。

Hybrid vector DB adapter

Milvus/Zilliz collection 支持 dense vector、BM25 sparse vector 和 RRF reranking。

Snapshot / Sync

本地 snapshot、Merkle file hashes、后台同步、trigger watcher 和跨进程锁共同支撑持续索引。

Evaluation harness

`evaluation/` 给出 SWE-bench 检索效率对比，是采用方复现实验的起点。

06

采用前确认与证据边界

采用前确认

  • 代码片段能否进入外部 embedding provider；若不能，先验证 Ollama 路径。
  • Milvus/Zilliz 的 collection limit、延迟、权限、清理和备份策略。
  • 同一仓库必须使用一致绝对路径，否则索引身份会分裂。
  • CI 目前更偏 build gate，lint 被注释，测试文件较少；生产引入要补自己的检索回归集。

证据与边界

  • 证据：README/docs、`packages/core`、`packages/mcp`、package manifests、CI/release、evaluation、LICENSE、GitHub metadata。
  • 边界：静态分析；未启动 MCP，未索引真实仓库，未复现 README 的效率实验。
  • 结论：适合作为大仓库 agentic coding 检索层试点，不应跳过内部任务集验证。
