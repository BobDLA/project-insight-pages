OpenHuman

  • OpenHuman 的核心不是“又一个聊天 agent”，而是 context first, conversation second：先同步并压缩个人数字生活，再让对话从本地上下文开始。

  • Zread 抓到了关键机制，但本报告以 README、CLAUDE.md 和当前源码核验：sidecar 与 React 18 说法已按源码修正。

01

新用户先看什么

适合谁、问题、差异、时机和最小验证。

适合谁

  • 想要桌面优先个人 AI agent 的技术用户。
  • 愿意用测试账号验证 Gmail、Slack、Calendar、GitHub、Notion 等连接源的团队或个人。
  • 研究 agent harness、技能系统、本地记忆和 Tauri/Rust 桌面架构的开发者。

解决什么问题

  • 把个人 agent 从 conversation first 推向 context first，减少每次从空白聊天开始的摩擦。
  • 把连接器、工具目录、模型路由、TokenJuice、语音、通知和桌面入口放入一个产品。
  • 让 agent 进入真实工作现场：meeting agent 可作为 Google Meet 参会者听会、转写、调用工具并通过 TTS/camera 回到会议。

和别的方案哪里不同

  • 差异在闭环：auto-fetch -> Memory Tree/Obsidian vault -> agent turn 召回。
  • 连接数据被 canonicalize 成 ≤3k-token chunks，scored，并进入层次 summary trees。
  • ControllerSchema 统一 CLI/RPC domain contract；Meeting Agent 则把 STT、diarization、Memory Tree、工具调用、TTS 和 lip-sync camera 连成实时协作链。

为什么现在值得看

  • 2026-02-18 创建后在 2026-05-18 仍活跃 push，早期关注度高。
  • 已有 CI、测试矩阵、release workflows 和多平台包装。
  • 代表个人 agent 从“聊天插件”走向“本地上下文中枢”的方向。

最小验证方式

  • 先用测试账号安装桌面端，不直接接主邮箱。
  • 连接一个低风险集成，触发一次 auto-fetch，并检查本地 memory/vault 产物。
  • 复现 README 的 Gmail 查询场景，观察工具状态、摘要质量、撤销连接和错误提示。

当前判断

  • 产品方向强，工程资产多。
  • 静态证据足以说明架构野心，但不足以证明隐私边界和端到端稳定性。
  • Zread 的机制线索有价值，但 sidecar、React 版本等说法需按当前源码修正。
02

核心创新提取

先判断项目拒绝了什么默认假设，再看哪些具名机制让它成立。

范式反转

OpenHuman 拒绝“agent 必须先聊天再慢慢积累上下文”的默认假设，改成 context first, conversation second：先通过连接源同步和压缩建立本地上下文，再让对话使用它。

Memory Tree

README.md 与 src/openhuman/memory/README.md 支撑：20 分钟 auto-fetch、≤3k-token Markdown chunks、SQLite/FTS/vector/graph、three concentric trees 和 Obsidian vault。

Intelligence Layer

Model routing 负责选模型；TokenJuice 先按 builtin/user/project 三层 rule overlay 对 tool output 做 classify -> match rule -> reduce，再进入 LLM context；subconscious/cron 承担后台任务。

Controller System

src/core/mod.rs 定义 ControllerSchema，src/core/all.rs 汇总 registry，dispatch.rs 校验并路由 RPC；这是 transport-agnostic domain contract。

CEF scanner 边界

CLAUDE.md 要求 migrated provider webviews 不新增 JS injection，采集和观测转向 native CDP scanner、Network/Page/Input/IndexedDB 等侧能力。

Mascot / Meeting / Voice

gitbooks/features/mascot/meeting-agents.md 支撑：mascot 作为 Meet 真实参会者，streaming STT 听会，diarization/postprocess transcript，TTS 注入 outbound mic，canvas/lip-sync 作为 outbound camera，并把会议写回 Memory Tree。

外部导览核验

Zread 的五个机制线索被吸收；但其 sidecar process 与 React 18 说法按当前 CLAUDE.md、app/package.json 修正。

架构视角

复杂项目，但报告选择裁剪版 C4 + 核心交互序列，避免把 4+1 强行拆成无证据文本卡。

Architecture Decision

桌面 shell、Rust core、记忆/工具/连接器、ControllerSchema 和后端 broker 的组合系统

选用的架构描述框架：裁剪版 C4 L1/L2 + 核心交互序列。Omitted views: 未绘制线程级并发、完整数据库 schema、生产后端拓扑和移动端/web 目标；当前证据重点支持 runtime boundary、transport contract 和 data pipeline。
Project complexity高：Tauri/CEF、Rust core、React UI、ControllerSchema registry、Memory Tree、OAuth/backend、语音和多渠道工具系统。
Selected frameworkC4 L1 Context + C4 L2 Container + Dynamic/Sequence。
Tailoring reason当前证据更适合说明容器边界、请求链、data pipeline 和 transport contract，而不是全量 4+1。
Deployment boundary桌面端、本地 workspace、OpenHuman backend、模型提供商、第三方 SaaS。

采用前确认与证据边界

采用前确认

  • 抓包或日志验证真实数据何时留在本机、何时经过 backend、LLM provider、OAuth proxy 或 TTS/search broker。
  • 确认 OAuth scope、rate limit、自动同步间隔、撤销后的停止同步和本地残留数据处理。
  • 检查 Memory Tree chunk、summary、dedupe、score、retrieval 是否可解释且可追溯。
  • 比较 TokenJuice 压缩前后的信息损失、token 降低、延迟和回答质量。
  • 验证 Meeting Agent 的参会者同意、音视频设备路由、STT/TTS 延迟、打断策略、transcript 存储位置和删除路径。
  • 验证 migrated provider 是否真的没有新增 JS injection，并评估 CDP scanner 权限/ToS 风险。
  • 注意 GPL-3.0 对闭源集成和二次分发的影响。

  • README/docs 支撑产品定位、context-first thesis、Memory Tree、TokenJuice、Meeting Agent/Voice、隐私叙事和安装路径。
