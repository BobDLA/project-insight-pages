# Project Insight Framework v2

## 目标

用纯静态或轻验证方式分析一个开源/内部项目，判断它是否值得关注、试点、采用或投入。报告必须同时服务两类读者：决策者能快速看懂价值，工程人员能追溯判断依据。

## 一级结构

1. 项目价值判断
2. Gold Example / Demo
3. 核心业务流程与关键资产
4. 技术实现与可落地性
5. 风险、壁垒与最终建议

## v2 改进点

### 证据协议

每个关键判断都必须绑定证据。证据来源分为：

- `README/docs`：项目自述、官方文档、架构文档
- `code`：源码、入口、接口、核心模块
- `config`：manifest、依赖、环境变量、CI、构建脚本
- `repo-meta`：GitHub API 元数据、最近提交、Star/Fork/Issue
- `license`：LICENSE、商业许可、贡献协议
- `static-inference`：基于以上证据做出的静态推断

规则：

- 能直接证明的结论使用 `README/docs`、`code`、`config`、`repo-meta`。
- 无法直接证明但合理推断的结论必须标为 `static-inference`。
- 纯静态分析不得声称项目已跑通。

### 项目类型适配

不同项目使用同一框架，但内容重点要适配项目类型：

| 项目类型 | Gold Example 重点 | 架构重点 | 工程化重点 |
| --- | --- | --- | --- |
| API/代理工具 | 请求/响应链路、协议兼容、provider 路由 | 服务入口、路由、适配器、配置 | 测试、鉴权、限流、错误映射、CI |
| 桌面/大型应用 | 用户工作流、数据接入、交互体验 | UI、服务、数据层、嵌入式运行时 | 构建、安装、发布、测试覆盖、许可证 |
| 文档/skill 项目 | 指令如何改变 agent 行为 | 文件分发形态、规则/skill/plugin 映射 | 版本、安装路径、一致性、示例质量 |

### 静态 Demo 规则

若执行边界是纯静态：

- `Demo 状态` 写为 `静态推演，未运行`。
- 给出“理想 Gold Example”和“静态证据支持”。
- 不生成真实运行结果，不伪造日志、截图或输出。
- 明确说明需要哪些条件才能完成真实验证。

### 质量门禁

通过标准：

- 不输出用户可见的项目分数。
- 三份报告的核心结构一致，但允许根据项目类型调整内容重心。
- 关键判断至少有一个证据来源。
- 静态 Demo 不伪装成已运行结果。
- 新用户能快速理解使用场景、价值和差异点。
- 风险、CI/CD、壁垒等交付信息只保留会影响采用决策的部分。

## HTML-first 交付标准

最终交付不应只是 Markdown 审计记录。除非用户明确只要纯文本，项目洞察分析应优先生成一个可阅读的 HTML 项目页。

HTML 报告默认只输出一个响应式页面。不要同时维护完整报告版和独立信息图版，除非用户明确要求第二个交付物。

HTML 报告默认包含：

- 首屏：项目名、一句话定位、采用口径、关键事实和冷结论。
- 导航默认面向首次了解的技术人员：Lean 判断、Gold Example / Demo、项目机制图、架构视角、核心资产、采用前确认；其中项目机制图和架构视角是可选模块。
- 默认不展示分数、总分或横向评分条；改用非评分事实和文字判断。
- 项目价值判断：保留并增强 Lean Canvas，回答适合谁、解决什么问题、差异点、为什么现在值得看、最小验证方式。
- Gold Example：静态或真实演示链路，必须明确是否运行；真实图片/视频只有在直接展示核心产品界面、核心工作流、用户动作或输出结果时才优先。
- 项目机制图：按项目实际从 UML Sequence/Interaction、UML Component/Logical、CLD、SFD、BOT、混合图中选择合适图型；若图表会空泛，可以不用。
- 架构视角：先做复杂度评估，再决定使用 C4、4+1、UML 或轻量替代视角；简单/中等项目优先 C4，复杂/异构项目才使用 4+1，并且核心交互图优先。
- 核心资产与价值：展示代码资产、规则资产、prompt-like 资产、测试资产、数据资产。
- 风险、壁垒与最终建议：说明什么时候项目价值会下降，给出最终决策。
- 证据边界：列出证据来源和未验证内容。

Markdown 报告可以作为 HTML 的可审计文本版，但不能替代最终洞察页。

## HTML 排版原则

参考 `ui-ux-pro-max-skill` 的 UI/UX 规则，来源记录见 `docs/reference/ui-design-references.md` 和 <https://github.com/nextlevelbuilder/ui-ux-pro-max-skill>。项目洞察页采用克制的 Minimalism / Swiss 报告风格：

- 首屏服务决策，不做过高的空白海报区。
- 使用稳定桌面容器、网格分区和 4/8px 间距体系。
- 正文不低于 16px，行高控制在 1.5-1.75；长段落限制在约 65-75 字符行宽。
- 通过字号、字重、间距和对比建立层级，不依赖装饰色。
- section 标题、说明和正文左边界必须统一；不要用大号左侧章节栏挤压正文。
- 图表 header 里的选择理由、场景和图型标签必须左对齐顺排，避免一侧空白、一侧内容过窄。
- 使用语义色 token：正文、弱文本、边框、表面、强调、成功、警告、危险。
- 使用统一 elevation / shadow scale，避免随机阴影。
- 导航触控高度不低于 44px。
- 移动端不得出现页面级横向滚动；宽表格必须使用横向滚动容器或改成卡片。
- 若加入动效，必须支持 `prefers-reduced-motion`。

## 输出原则

Lean Canvas、C4、4+1、UML、系统动力学和工程化检查都是分析支架，不拆成额外一级模块，也不强制每份报告都使用。Lean Canvas 放入项目价值判断；业务流程和资产放入核心业务流程与关键资产；C4 / 4+1、运行机制和工程化检查放入技术实现与可落地性；最终建议负责给出决策。

架构视角采用自适应裁剪：

- 简单/中等项目：首选 C4，输出 L1 Context、L2 Container 和核心 Dynamic/Sequence。
- 复杂/异构项目：使用 4+1 做理论分类，并用 C4/UML/Mermaid 表达场景、过程、开发/实现和部署视图。
- 规则包、prompt/skill、小工具：可使用 C4-light、规则执行链路或分发结构；不要强行套 4+1。
- 无论选择哪种框架，都优先输出核心业务流程交互图。
- 架构 section 不能只是贴 Logical/Process/Physical 标题的文字卡片；必须把 Mermaid 或等价结构化图源渲染为可读图形，并保留源码。
- 手机端若完整图形不可读，使用纵向步骤 fallback，不产生页面级横向滚动。
- 不把所有 Mermaid 都交给通用自动排图。按项目语义选择图法：规则/skill 项目用行为约束与分发结构，API/proxy 项目用请求边界与适配链，桌面/大型应用用工作流边界、数据总线和部署边界。

若面向首次接触项目的技术读者生成 HTML，可把标题改写为更直观的用户路径：`新用户先看什么`、`Gold Example / Demo`、`项目机制图`、`架构视角`、`核心资产与价值`、`采用前确认与证据边界`。这是展示层重排；其中 `项目机制图` 和 `架构视角` 可按项目省略。

图表选择规则：

| 图型 | 适用场景 |
| --- | --- |
| UML Sequence / Interaction | 解释一次请求、任务或用户工作流 |
| UML Component / Logical | 解释模块、适配器、数据层或分发文件关系 |
| CLD | 解释采用、信任、兼容覆盖、风险控制等反馈回路 |
| SFD | 解释上下文、缓存、未完成需求、技能、数据等存量/流量 |
| BOT | 解释随时间变化的概念趋势；若无实测数据必须标注概念趋势 |

UML+CLD 混合图只是一个可选示例，不是默认模板。
对于确实需要同时说明“微观执行链路”和“宏观反馈/存量变化”的系统，可以参考 `docs/reference/image.md` 使用 UML + 系统动力学混合图：中心放一次具体任务或请求，外圈只叠加有解释增益的 CLD/SFD/BOT。

Gold Example 的真实媒体选择必须先过相关性门槛：不能因为图片来自仓库就使用。若图片只是旁支集成、品牌图、吉祥物、无关客户端界面或生态截图，且不能帮助新用户理解使用场景、价值或差异点，应降级为 README/example 命令流、before/after 示例或结构化场景卡片。
