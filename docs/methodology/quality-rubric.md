# 报告质量检查清单

本清单用于迭代修正，不输出为项目打分。

## 必须满足

- 不出现项目分数、总分、综合评分或评分条。
- 只输出一个响应式 HTML，除非用户明确要求第二个信息图或截图版。
- 首屏和第一段正文能让新用户看懂：适合谁、解决什么问题、和别的方案哪里不同。
- Lean 内容必须保留并增强：为什么现在值得看、最小验证方式不能缺失。
- Gold Example 优先使用相关的真实项目图片/视频；只有直接展示核心产品界面、核心工作流、用户动作或输出结果时才算相关。否则使用仓库 example 或结构化场景卡片。
- 静态分析必须明确标注未运行，不伪造日志、截图或结果。
- 项目机制图是可选组件；必须按实际项目选择 UML / CLD / SFD / BOT / 混合图，不硬套 UML+CLD。
- 架构视角必须先说明复杂度评估、框架选择、裁剪理由和省略内容；简单/中等项目优先 C4，复杂/异构项目才使用 4+1。
- 不管采用 C4 还是 4+1，都必须优先提供核心业务流程交互图；如果没有，必须说明不适合的理由。
- 可视化图表必须画成可读的流程/组件/系统动力学画布，不能只是卡片拼接或未对齐箭头。
- section 标题、说明、图表 header 和正文左边界必须统一，不出现左侧空列或右侧内容被挤小。
- 桌面、1080px 竖向截图和手机宽度均不得出现页面级横向溢出、图表裁切或标题异常断字。
- 架构 section 不能只是贴 Logical/Process/Physical 标题的文字卡片，也不能只显示 Mermaid 源码；必须渲染为可读图形并保留源码。
- 架构图必须匹配项目语义，不能把规则包、API 代理、桌面工作台都画成同一种通用节点网络。
- 核心资产必须绑定到 README/docs、源码、配置、示例或明确的静态推断。
- 采用前确认只保留会影响试用/采用决策的内容，不让 CI/CD、风险、壁垒喧宾夺主。
- 证据与边界能区分 README/docs、code、config、repo-meta、license、static-inference。

## 失败条件

- 报告仍含具体分数、`总分`、`综合评分`、`评分口径` 等输出。
- 仍然维护重复的完整报告和独立信息图 HTML。
- 大段文字未拆成条目或流程，导致扫描困难。
- Demo 无来源、不可追溯，或把静态推演写成真实运行。
- Gold Example 使用了与核心项目价值无关的仓库图片，例如旁支集成、品牌图、无关客户端界面或生态截图。
- 图表与项目实际机制不匹配，或把示例流程写成固定模板。
- 架构视角没有复杂度判断，或者把简单项目强行套成 4+1。
- 图表文字太小、节点被裁切、箭头不连贯，或手机端没有可读 fallback。
- 对规则包、小工具、prompt/skill 仓库等强行套 4+1，导致 Logical/Physical/Process 等视角空泛。
- 对 API/proxy、桌面应用、文档/skill 项目使用同一套不加调整的重点。
- 新用户读完仍不知道使用场景、核心价值或与替代方案的区别。

## 迭代顺序

1. 删除评分表达，改为文字采用口径和非评分事实。
2. 补齐新用户关心的使用场景、价值、差异点。
3. 用相关真实媒体或 example 强化 Gold Example；无法说明相关性时改成流程卡片。
4. 按项目机制选择图型，画出可读图表，并保留 Mermaid / 结构化图源。
5. 将长段落拆成 bullet、label/value 或机制图。
6. 精简交付审计内容，只保留影响采用的提醒。
7. 补证据边界和未验证条件。
